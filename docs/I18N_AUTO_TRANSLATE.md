# I18N Auto-Translate — как новые статьи переводятся сами (дизайн)

> Простыми словами: **English = оригинал, пишется ОДИН раз. Перевод на другие языки —
> отдельный автоматический шаг.** Автор/скрипт пишет только английский; машина размножает
> на активные локали. Это план будущего «авто-шага» (Phase 4) — пока перевод гоняется руками
> пачками. Контекст: `docs/I18N_IMPLEMENTATION_PLAN.md`.

## Поток целиком

```
1. Контент создаётся (генерация статьи скриптом / новость через Worker)
        → пишется в Supabase (таблицы articles / news)            [как сейчас]
2. [НОВЫЙ АВТО-ШАГ] Переводчик берёт ещё-не-переведённые опубликованные строки
        для активных локалей → GPT-4o-mini + глоссарий → кладёт в *_translations
3. Сборка Astro генерит /es/, /fr/, /zh/ страницы из *_translations  [как для 6 пилотных]
```

С точки зрения автора — **ничего не меняется**: одна английская статья. Шаги 2-3 — фоновые.

## Как именно запускается (два варианта, можно вместе)

- **A. Ежедневный cron** (GitHub Action, как существующие воркфлоу) — раз в день гоняет
  `scripts/i18n/translate-content.mjs` по всем типам/локалям. **Идемпотентно** — берёт только
  новое (что ещё не переведено). Простой, надёжный, развязан с генерацией. ← **основной способ**.
- **B. Хук в генерацию** — сразу после вставки статьи скрипт вызывает перевод (статья появляется
  на всех языках в тот же день, без ожидания cron). Опционально, для немедленности.

«Идемпотентно» = даже если запустить 10 раз, каждая вещь переведётся один раз (уже переведённое
пропускается). Поэтому ежедневный cron безопасен.

## Защита (guardrails)

- **Перевод НЕ ломает английский.** Это отдельный шаг; английская статья уже опубликована.
  Упал перевод — английская версия живёт как ни в чём не бывало.
- **Только активные локали** (`I18N_LOCALES` / shipped-default). Выключил локаль → по ней не переводит.
- **Cost-guard** (`--max-cost`), идемпотентность, **sanitize на запись** [P1], битый HTML →
  `status='review'` (не публикуется, пока человек не проверит).
- **Стоимость** — доли цента на статью на все языки.

## Особый случай: новости

News отдаются через Worker/API, а не напрямую из Supabase-сборки. Их авто-перевод требует
доработки Worker (отдавать локализованный JSON) — отдельный кусок (block B в плане). Поэтому
порядок: сначала авто-перевод для обычных **статей** (articles), потом **news / players**.

## Когда включаем

1. **НЕ сейчас.** Сначала es-пилот (6 страниц) проходит GSC-окно (2-3 нед).
2. Зелёный свет → включаем авто-перевод для **es** (ежедневный cron по articles).
3. Дальше добавляем **fr**, потом **zh** в тот же cron — по мере запуска каждого языка.
4. news / players — после доработки Worker (block B).

## Откат

Выключить локаль (`I18N_LOCALES=none` или убрать язык из списка) или отключить cron
(репо-переменной/disable — как мы сделали с Indexing API). Английский контент от этого не зависит.

## Итог одной фразой

Пишешь английскую статью один раз → ежедневный фоновый cron сам переводит её на активные языки →
на следующей сборке появляются `/es/`, `/fr/`, `/zh/`. Включаем после зелёного GSC по пилоту.
